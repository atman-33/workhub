team-ops — Team Operations Plugin for the workhub Marketplace

1Concept & principles

What this plugin is, and the rules every later decision follows from.

team-ops turns one team-shared folder (Google Drive / OneDrive / file server — anything synced to a local path) into the team's single source of truth for knowledge and project operations, maintained jointly by humans and AI agents.

PrincipleConsequence
Shared folder = SSoT Backlog, sprints, specs, and knowledge live as Markdown in the shared folder. External tools (monday.com etc.) are future optional mirrors, never the master.
Readable by newcomers Everything is plain Markdown with _index.md catalogs. A new member (or an AI session) can self-serve answers by reading knowledge/ and docs/spec/spec.md.
Multi-person safe One item = one file (sync-conflict resistant), ID conventions, and an activity-log.md audit trail of AI writes.
Storage-agnostic Config holds one local path (teamRootPath). No Drive/OneDrive API coupling; never place a .git directory inside the shared folder.
Token-free data layer Repo sync and progress snapshots are plain Node scripts. AI runs only where judgment is needed (report narrative, spec update) — a daily routine stays cheap.
Knowledge accumulates naturally A SessionStart hook injects a standing norm: "when you discover reusable team knowledge, propose saving it via team-kb-save". Capture happens at the moment of discovery, not in a weekly chore.

2Shared folder layout

The ai/ zone inside the team-shared folder. Color = who owns the writes.

human-curated (AI appends/proposes) AI-maintained (humans read) machine-generated (scripts, no AI tokens) 
<teamRoot>/ai/
  knowledge/                    # team knowledge base — the "read this and you understand us" zone
    _index.md                   # whole-KB catalog (team-kb-index maintains)
    rules/                      # team rules & working agreements
    onboarding/                 # newcomer guides
    <topic>/                    # kebab-case topic folders, each with _index.md
  projects/<project>/           # one folder per project (generalized from the retired scrum plugin's Epic folder)
    _index.md                   # project catalog + current-state summary
    config/
      project.json              # shared project settings: repos[], sprint length, point scale
    prd/                        # product requirements
    backlog/
      product-backlog.md        # ordered PBL overview (id / title / status / points / sprint)
      items/<id>-<slug>.md      # PBI: user story, acceptance criteria, tasks (1 item = 1 file)
      progress-history.jsonl    # daily snapshot: date, status counts, remaining points
    sprints/<yyyy.nn>/
      planning.md               # sprint goal + committed items (plan-sprint)
      scope.json                # machine-readable sprint scope: item ids + points (burndown baseline)
      review.md  retro.md       # review-sprint / write-retro
    docs/
      decisions/                # ADR-style decision records
      spec/spec.md              # living spec — updated daily from PBIs + merged diffs (update-spec)
    reports/
      daily/<date>.html         # daily progress report + burndown (report-daily-progress)
    repos/<repo-name>/          # per-repository tracking data (multi-repo aware)
      repo-state.json           # url, devMainBranch, lastSyncedSha, lastSyncAt
      commits.jsonl             # dev-main-branch commits, append-only incremental
      diff-vs-default.json      # dev-main vs default branch: ahead/behind, diffstat, files
      pbi-activity.json         # per-PBI: merged commits, files, +/- lines, last merge date
  _meta/
    conventions.md              # naming / ID / branch conventions the AI must follow
    activity-log.md             # append-only log of AI writes (audit for multi-person use)
PBI frontmatter (schema draft): id (e.g. P-0012), title, status (todo | doing | review | done), points, sprint, assignee, repos (which repos it touches), created, updated. Body: ## Story / ## Acceptance Criteria / ## Tasks / ## Notes.

3Configuration — two layers

Shared team settings live in the shared folder; machine-local paths stay local.

Local: .claude/team-context.json per machine

Created by setup-team-context in each working project / harness workspace. Gitignored (mirrors the project-context.json pattern).

{
  "teamRootPath": "G:\\共有ドライブ\\team-x",
  "repoWorkspacesRoot": "C:\\repos\\.team-ops-repos",
  "me": "atman",
  "activeProjects": ["app-renewal"]
}

Shared: projects/<p>/config/project.json per team

Lives in the shared folder — every member and every AI session sees the same repo definitions and sprint settings.

{
  "repos": [
    { "name": "app",
      "url": "https://github.com/org/app",
      "devMainBranch": "develop/app-renewal",
      "defaultBranch": "main" },
    { "name": "api",
      "url": "https://github.com/org/api",
      "devMainBranch": "develop/app-renewal" }
  ],
  "sprint": { "lengthDays": 10, "pointScale": [1,2,3,5,8,13] }
}
Clone policy: script-owned read-only mirrors under repoWorkspacesRoot (default ~/.team-ops-repos/) — never inside the shared folder, never your own development checkout.
Team-wide shared config: _meta/team.json — settings that apply to the whole team regardless of project, currently the content language (BCP 47 tag, e.g. "ja"): the language KB, backlog, sprint, and spec content is written in. Injected into every session as <content-language>. Plugin docs and skills themselves stay English.

7PBI ↔ code linkage & burndown mechanics

Linkage (convention-based)

  1. start-pbi sets the PBI to doing and issues the branch name pbi/<id>-<slug> — the convention is enforced where work begins.
  2. Commits/PRs merged into the project's dev-main branch carry the PBI id (branch name in the merge commit, or [P-0012] in the message).
  3. sync-project-repos.mjs extracts PBI ids from new commits and aggregates per-PBI: merged commit count, files touched, +/- lines, last merge date → repos/<repo>/pbi-activity.json.
  4. Open a PBI file → its id keys into pbi-activity.json across all repos: you see how far implementation has progressed and where.
Works with any git hosting — it only reads the cloned mirror; no GitHub/GitLab API needed for the core.

Burndown

  1. plan-sprint writes sprints/<id>/scope.json — committed item ids and their points: the baseline.
  2. The daily snapshot appends remaining points (sum of points of not-done scope items) to progress-history.jsonl.
  3. report-daily-progress draws: ideal line (baseline → 0 over sprint days) vs actual line (daily remaining), plus scope-change annotations if items were added/removed mid-sprint.
Status changes are made in the PBI frontmatter (by humans in the editor, or via skills) — the snapshot only reads, so the chart is correct regardless of who edited what, and needs no central server.

8Relationship to existing plugins

PluginRelationship
scrum Retired and removed from the marketplace (2026-07-20, owner decision). team-ops is its zero-base successor around the shared-folder-as-SSoT idea; the monday.com + Drive snapshot workflow was not carried over — external tools may return later as optional mirrors.
workhub (vault) Different scope: the vault is personal task/knowledge space; team-ops is the team's shared space. The KB skills mirror the vault's kb-* philosophy (indexes, zones) but operate on <teamRoot>/ai/.
engineering Unchanged. team-ops' knowledge-capture norm complements capture-rule: repo-specific rules still go to the repo; team-process knowledge goes to the team KB.

Marketplace placement: optional · project scope (needs per-project config), registered in marketplace.json + docs/plugins.md. All scripts are Node ESM (.mjs) per plugin-authoring rules.

9Decision record (owner review, 2026-07-20)

  1. Folder layout (§2) — approved as designed (top folder name ai/).
  2. PBI id format — P-0001 per project, sequential.
  3. Sprint folder naming — sprints/<year>.<sequence>/ (e.g. 2026.15).
  4. Daily routine trigger — scheduled Claude routine on the owner's machine first; team-shared automation (CI) is a later step.
  5. spec.md granularity — one spec.md per project to start, splitting into spec/<area>.md when it grows (update-spec proposes the split).
  6. Language — team content language is configurable in the shared _meta/team.json (language); plugin docs/skills stay English.
  7. This document — lives at plugins/team-ops/docs/design.html; the repo rule .claude/rules/team-ops-design-doc.md requires updating it alongside any behavioral plugin change.
